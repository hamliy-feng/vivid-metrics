# Codex 前端页面固定流程

## 这份文档现在怎么用

这份文档不是让你先写一堆专业说明。

它现在只做一件事：

- 规定 Codex 在 `vibe coding` 时，应该先按什么顺序做

也就是：

- 你继续说自然语言
- Codex 自己先读统一标准
- Codex 自己选合适组件
- Codex 自己按固定样式落页面
- 最后再自己检查有没有跑偏

这才更适合你的工作方式。

---

## 默认模式：自然语言直接做

以后默认先读这份：

- `.codex-memory/spec/frontend-design-standards.md`

你只需要告诉 Codex：

- 这个页面是干什么的
- 主要功能是什么
- 特别不想要什么效果

其他事情默认由 Codex 自己完成：

- 组件从 `shadcn/ui` 里选
- 颜色、字号、圆角、间距按统一标准走
- 页面骨架按产品页方式组织

不要再把“先写组件清单”当默认前置条件。

---

## 增强模式：复杂页面再补说明

只有当页面特别复杂，或者你对结构有很强要求时，才建议再补材料。

可补材料包括：

- 团队自己的视觉规范文档
- 一段需求补充说明
- 你自己写的简单结构说明

但这些都不是默认必填。

---

## 固定执行顺序

以后 Codex 做页面，默认按这个顺序：

1. 先读 `Codex Frontend Design Standards.md`
2. 先根据自然语言理解页面目标
3. 先输出本页定位
4. 先输出页面骨架
5. 再自己选 `shadcn/ui` 组件
6. 再开始实现页面
7. 最后按统一标准自检

这里最关键的点是：

- 用户不需要先写组件清单
- 组件应该由 Codex 在统一标准里自己选
- 用户只需要说业务目标和功能需求

---

## 固定底座

1. 组件统一从 `shadcn/ui` 来
2. 风格统一按项目规则和统一标准走
3. 如果你还有团队自己的视觉规范，可作为补充参考
4. 如果需求没写清楚，不要自己发明新风格，先沿用现有规则
5. 如果项目本身不是 `React / Next.js + Tailwind` 这条路线，就先停下来说明，不能硬套 `shadcn/ui`

最后这一条不能糊弄过去。

如果技术路线不对，硬上只会让页面更乱，不会更统一。

---

## Codex 回答时的默认顺序

以后更适合的顺序是：

### A. 本页定位

- 这是什么页面
- 最重要的动作是什么

### B. 页面骨架

- 顶部有什么
- 中间分几块
- 主要内容怎么排

### C. 组件选择

- 准备使用哪些 `shadcn/ui` 组件
- 为什么这样选

### D. 视觉约束

- 颜色怎么用
- 字号怎么用
- 圆角怎么用
- 间距怎么用
- 哪些效果不能出现

### E. 页面实现

- 开始真正写代码

### F. 自检结果

- 有没有偏离统一标准
- 有没有混入第二套组件库
- 有没有出现不规律字号、间距、圆角

---

## 最简单的调用方式

以后直接这样说就行：

```text
先阅读 `.codex-memory/spec/frontend-design-standards.md`。
然后按这份标准直接设计并实现页面。

页面需求：
[直接用自然语言写]

除非我明确要求，否则：
- 默认使用 shadcn/ui
- 不要更换组件库
- 不要更换主题
- 不要自由发挥成营销站风格
```

---

## 最后结论

以后真正稳定的方式不是：

- 让你先写一堆专业信息

而是：

- 你继续用自然语言提需求
- Codex 自己先读统一标准
- Codex 自己在同一套组件和样式规则里完成页面

这才更符合你要的 `vibe coding`。
